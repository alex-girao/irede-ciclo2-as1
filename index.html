O que é Programação Orientada a Objetos? Paradigma

A POO é um paradigma de programação que se baseia no conceito de "objetos" para estruturar e organizar o código.

Cada objeto possui características (atributos) e comportamentos (métodos), definidos por classes — que funcionam como moldes para a criação de objetos.

Com a POO, conseguimos modelar o mundo real dentro do código.

Objetos Classes Atributos Métodos
Classes e Objetos Conceito Central

Classe: estrutura que define as características (atributos) e comportamentos (métodos) de um tipo de objeto. É como um molde ou projeto.

Objeto: entidade concreta, criada a partir de uma classe. Cada objeto tem seus próprios valores de atributos.

Exemplo — um carro possui atributos (cor, modelo, ano) e comportamentos (acelerar, frear, buzinar).

CLASSE — O MOLDE
public class Carro { String cor; String modelo; int ano; void buzinar() { System.out .println("Bii Bii!"); } }
OBJETO — A INSTÂNCIA
Carro meuCarro = new Carro(); meuCarro.cor = "Vermelho"; meuCarro.modelo = "Sedan"; meuCarro.ano = 2022; meuCarro.buzinar();
Encapsulamento Modificadores de Acesso

O encapsulamento é a prática de ocultar os detalhes internos de uma classe, expondo apenas o que é necessário para quem a utiliza.

Por que usar? Proteção dos dados · Manutenção facilitada · Controle de acesso.

Getters e Setters Acesso Controlado

No encapsulamento, os atributos da classe são privados — não podem ser acessados diretamente de fora da classe.

Para permitir a leitura ou modificação desses atributos de maneira segura, usamos Getters (acessar) e Setters (modificar).

public class Conta { private double saldo; // atributo encapsulado // Getter — retorna o saldo atual public double getSaldo() { return saldo; } // Setter — valida antes de modificar public void setSaldo(double saldo) { if (saldo >= 0) { this.saldo = saldo; } else { System.out.println("Erro: saldo não pode ser negativo!"); } } }
Construtores Inicialização

Um construtor é um método especial chamado automaticamente quando um objeto é criado. Ele serve para inicializar os atributos da classe.

✓ O nome do construtor deve ser igual ao nome da classe
✓ Não possui tipo de retorno (nem void, nem outro tipo)
✓ Pode receber parâmetros para definir valores iniciais
public class Pessoa { String nome; int idade; // Construtor da classe Pessoa public Pessoa(String nome, int idade) { this.nome = nome; // "this" refere-se ao atributo da classe this.idade = idade; } void apresentar() { System.out.println("Olá, meu nome é " + nome + " e tenho " + idade + " anos."); } } // Usando o construtor: Pessoa p1 = new Pessoa("Carlos", 25); Pessoa p2 = new Pessoa("Ana", 30); p1.apresentar(); // Olá, meu nome é Carlos e tenho 25 anos. p2.apresentar(); // Olá, meu nome é Ana e tenho 30 anos.
Sobrecarga de Construtores Overloading

A sobrecarga permite que uma classe tenha múltiplos construtores com o mesmo nome, desde que tenham diferentes listas de parâmetros.

Isso significa que podemos criar objetos de diferentes maneiras!

// Construtor 1 — nome + idade public Pessoa(String nome, int idade) { this.nome = nome; this.idade = idade; } // Construtor 2 — apenas nome public Pessoa(String nome) { this.nome = nome; this.idade = 0; // idade padrão } // Construtor 3 — sem parâmetros public Pessoa() { this.nome = "Desconhecido"; this.idade = 0; } // Usando os 3 construtores: Pessoa p1 = new Pessoa("Carlos", 25); Pessoa p2 = new Pessoa("Ana"); Pessoa p3 = new Pessoa();

🚀 Curiosidades sobre POO
🏛
Os 4 Pilares da POO
A POO se sustenta em quatro pilares: Encapsulamento, Herança, Polimorfismo e Abstração. Nesta aula vimos o encapsulamento — os demais serão abordados nas próximas unidades.
🔑
A palavra-chave this
Dentro de um construtor ou método, this refere-se ao objeto atual. É usada para diferenciar atributos da classe de parâmetros com o mesmo nome: this.nome = nome;
🔧
Construtor padrão automático
Se você não declarar nenhum construtor, o Java cria automaticamente um construtor vazio sem parâmetros. Porém, se você declarar qualquer construtor explicitamente, o padrão não é criado automaticamente.
🏗
Objetos vivem no Heap
Ao usar new, o objeto é alocado na memória Heap. A variável que referencia o objeto fica na Stack. Por isso o operador == compara referências, não o conteúdo dos objetos — use equals() para comparar conteúdo!
📦
Tudo é (quase) objeto em Java
Com exceção dos tipos primitivos (int, double, char, boolean...), tudo em Java é um objeto. Inclusive String, que é uma classe completa — não um tipo primitivo!
🛡
A regra de ouro do Encapsulamento
Declare todos os atributos como private e forneça getters e setters públicos. Isso protege os dados de modificações inválidas e facilita futuras manutenções no código.

O que é Herança? Inheritance

A herança é um dos pilares da POO que permite que uma classe "herde" atributos e métodos de outra classe, promovendo o reuso de código.

A classe que herda é chamada de subclasse (ou classe filha), e a classe da qual ela herda é chamada de superclasse (ou classe pai).

Em Java, usamos a palavra-chave extends para declarar herança. A relação expressa um conceito de "é-um" (is-a): um Cachorro é-um Animal.

Herança extends Subclasse Superclasse
Usando extends Conceito Central

Com extends, a subclasse recebe automaticamente todos os atributos e métodos public e protected da superclasse, além de poder definir seus próprios membros.

SUPERCLASSE
public class Animal { private String nome; public Animal(String nome) { this.nome = nome; } public String getNome() { return nome; } public void emitirSom() { System.out.println("..."); } }
SUBCLASSE
public class Cachorro extends Animal { private String raca; public Cachorro( String nome, String raca) { super(nome); this.raca = raca; } public void buscarBola() { System.out.println( getNome() + " buscou!"); } }
A palavra-chave super Chamando o Pai

A palavra-chave super é usada dentro de uma subclasse para acessar membros da superclasse.

✓ super() — chama o construtor da superclasse (deve ser a 1ª instrução do construtor filho)
✓ super.metodo() — chama um método da superclasse (útil para estender o comportamento original)
public class Cachorro extends Animal { private String raca; public Cachorro(String nome, String raca) { super(nome); // 1º: chama Animal(nome) this.raca = raca; } public void emitirSom() { super.emitirSom(); // chama Animal.emitirSom() System.out.println("Au au!"); } }
Sobrescrita de Métodos @Override

A sobrescrita (override) permite que a subclasse redefina um método herdado da superclasse, alterando seu comportamento para o contexto específico da classe filha.

A anotação @Override avisa o compilador que estamos sobrescrevendo — ele detecta erros de digitação no nome do método.

NA SUPERCLASSE
// Animal.java public void emitirSom() { System.out .println("..."); } // saída: ...
NA SUBCLASSE
// Cachorro.java @Override public void emitirSom() { System.out .println("Au au!"); } // saída: Au au!
Hierarquia de Classes Class Tree

Quando classes herdam umas das outras forma-se uma hierarquia (árvore de classes). Uma subclasse pode ser superclasse de outra, criando múltiplos níveis.

Animal
↓
Mamifero

O operador instanceof verifica se um objeto pertence a uma determinada classe na hierarquia:

Cachorro c = new Cachorro("Rex", "Labrador"); System.out.println(c instanceof Cachorro); // true System.out.println(c instanceof Animal); // true (herança!) System.out.println(c instanceof Gato); // false

🚀 Curiosidades sobre Herança em Java
🚫
Java não tem herança múltipla de classes
Diferente de C++, Java só permite herdar de uma classe por vez. Isso evita o "Problema do Diamante". Para herança múltipla de comportamento, Java usa interfaces com implements — uma classe pode implementar várias interfaces!
📚
A classe Object tem 11 métodos
java.lang.Object define: equals(), hashCode(), toString(), getClass(), clone(), notify(), notifyAll(), três sobrecargas de wait() e finalize(). Todas as suas classes já os herdam automaticamente!
🔍
@Override previne bugs silenciosos
Se você escreve @Override e o nome do método não existe na superclasse, o compilador avisa imediatamente. Sem a anotação, um erro de digitação como emitirsom() (minúsculo) criaria um novo método em vez de sobrescrever — um bug difícil de encontrar!
🧱
final impede herança e sobrescrita
Uma classe final não pode ser estendida. Um método final não pode ser sobrescrito. A classe String em Java é final — por isso você não pode criar uma subclasse de String. Isso garante que o comportamento de String nunca seja alterado.
🎭
Upcasting é automático, downcasting precisa de cast
Atribuir um objeto filho a uma variável pai é automático: Animal a = new Cachorro("Rex","Lab"); — isso é upcasting. O caminho inverso (downcasting) requer cast explícito: Cachorro c = (Cachorro) a; e pode lançar ClassCastException se o objeto não for do tipo correto.
💬
toString() é chamado implicitamente
Quando você faz System.out.println(objeto), o Java chama objeto.toString() automaticamente! Sobrescreva toString() nas suas classes para ter logs úteis — caso contrário verá algo como Cachorro@7852e922 no console.

O que é Polimorfismo? 3º Pilar da POO

Poli = muitos · Morphos = formas. Polimorfismo é a capacidade de um objeto se comportar de diferentes maneiras dependendo do seu tipo real.

Na prática: o mesmo método chamado em referências diferentes produz resultados diferentes — sem precisar saber qual é o tipo exato do objeto.

Em Java existem dois tipos: Sobrescrita (runtime — via herança) e Sobrecarga (compile-time — na mesma classe).

Polimorfismo @Override Overload Dynamic Dispatch
Despacho Dinâmico Runtime

A JVM determina em tempo de execução qual versão do método deve ser chamada — baseada no tipo real do objeto, não no tipo da variável.

Isso é o coração do polimorfismo de herança em Java: o despacho dinâmico.

// Tipo da variável: Animal | Tipo real do objeto: Cachorro Animal a1 = new Cachorro("Rex"); Animal a2 = new Gato("Mia"); Animal a3 = new Passaro("Piu"); a1.emitirSom(); // → Au au! (Cachorro.emitirSom) a2.emitirSom(); // → Miau! (Gato.emitirSom) a3.emitirSom(); // → Piu piu! (Passaro.emitirSom)
Sobrescrita vs Sobrecarga Comparativo

São dois conceitos frequentemente confundidos — mas são completamente diferentes:

Sobrescrita (@Override)Sobrecarga (Overload)
NomeMesmoMesmo
ParâmetrosIguaisDiferentes
Onde ocorreSubclasse → SuperclasseMesma classe
QuandoRuntimeCompile-time
Anotação@Override—
Polimorfismo em Arrays Potência Real

Arrays e listas declarados com o tipo da superclasse podem armazenar objetos de qualquer subclasse. Ao iterar, cada objeto chama sua própria versão do método.

Isso torna o código extensível: adicionar um novo tipo de animal não exige mudar o loop!

Animal[] animais = { new Cachorro("Rex"), new Gato("Mia"), new Passaro("Piu") }; for (Animal a : animais) { a.emitirSom(); // cada um chama SUA versão! } // Rex diz: Au au! // Mia diz: Miau! // Piu diz: Piu piu!
Classes Abstratas abstract

Uma classe abstract não pode ser instanciada diretamente — existe apenas para ser estendida. Ela pode definir métodos abstract sem implementação, forçando as subclasses a implementá-los.

Perfeita para polimorfismo: garante que todos os tipos na hierarquia implementem os métodos necessários.

public abstract class Animal { private String nome; public Animal(String nome) { this.nome = nome; } public String getNome() { return nome; } // método abstrato: sem corpo, subclasse DEVE implementar public abstract void emitirSom(); } // Animal a = new Animal("X"); // ERRO! Classe abstrata Animal a = new Cachorro("Rex"); // OK — Cachorro implementa emitirSom()
Casting em Hierarquias Upcasting / Downcasting

Upcasting — subclasse → superclasse. Automático e seguro. Downcasting — superclasse → subclasse. Explícito e pode lançar ClassCastException.

Sempre use instanceof antes de fazer downcasting para garantir segurança.

// Upcasting — automático (sempre seguro) Animal a = new Cachorro("Rex"); // OK! // Downcasting — com verificação prévia (seguro) if (a instanceof Cachorro) { Cachorro c = (Cachorro) a; // OK! tipo real é Cachorro c.buscarBola(); // método específico de Cachorro } // Downcasting sem verificação — perigoso! Gato g = (Gato) a; // ClassCastException! a é Cachorro, não Gato

🚀 Curiosidades sobre Polimorfismo
⏱
Polimorfismo de tempo de execução vs compile-time
A sobrescrita (@Override) é resolvida em runtime — a JVM escolhe o método na hora da execução. A sobrecarga (overload) é resolvida em compile-time — o compilador escolhe qual versão chamar com base nos tipos dos argumentos.
📜
Princípio de Liskov (LSP)
O Princípio de Substituição de Liskov diz: qualquer subclasse deve poder ser usada no lugar da superclasse sem quebrar o programa. É a base do bom uso de polimorfismo — se Cachorro extends Animal, qualquer código que usa Animal deve funcionar com Cachorro.
🎭
toString() é polimorfismo!
System.out.println(objeto) chama objeto.toString() via polimorfismo! A JVM verifica o tipo real do objeto e chama a versão correta de toString() — seja de Object, de Veiculo, de Carro, ou outra. Todo println(obj) é despacho dinâmico em ação.
🦦
Abstract vs Interface
Classes abstratas e interfaces cumprem papéis parecidos mas diferentes. Uma classe abstrata pode ter atributos e métodos com implementação — modela uma hierarquia real ("é-um"). Uma interface define apenas um contrato de comportamento ("pode-fazer") e permite que classes não relacionadas a implementem.
🔮
Method Table (vtable)
Internamente, a JVM mantém uma tabela de métodos virtuais (vtable) para cada classe. Quando você chama um método por uma referência polimórfica, a JVM consulta essa tabela para encontrar a implementação correta — tudo acontece em nanossegundos!
🧩
Polimorfismo via Interfaces
Além de herança, Java permite polimorfismo via interfaces. Uma variável do tipo Runnable pode referenciar qualquer objeto que implemente Runnable. Isso permite polimorfismo entre classes sem hierarquia de herança comum — muito mais flexível!

Classes Abstratas abstract

Uma classe abstrata é um modelo incompleto que define estrutura e contrato para suas subclasses. Não pode ser instanciada diretamente com new — existe apenas para ser estendida.

Pode conter: atributos, construtores, métodos concretos (com implementação) e métodos abstratos (sem corpo, que subclasses devem implementar).

CLASSE ABSTRATA
public abstract class Forma { private String cor; public Forma(String cor) { this.cor = cor; } // método CONCRETO public String getCor() { return cor; } // método ABSTRATO public abstract double calcularArea(); }
SUBCLASSE CONCRETA
public class Circulo extends Forma { private double raio; public Circulo( String cor, double raio) { super(cor); this.raio = raio; } @Override public double calcularArea() { return Math.PI * raio * raio; } }
abstract método concreto método abstrato @Override
Associação usa-um

A Associação é a relação mais genérica entre classes. Indica que uma classe “usa” outra, sem possuí-la ou controlar seu ciclo de vida. A relação pode ser unidirecional ou bidirecional.

Exemplo: um Médico atende um Paciente. Ambos existem independentemente — o Médico não cria nem destrói o Paciente.

public class Medico { private String nome; // Associação: Médico usa Paciente via parâmetro public void consultar(Paciente p) { System.out.println( "Dr. " + nome + " atendendo " + p.getNome() ); } } // Médico e Paciente existem de forma completamente independente Medico m = new Medico("Dra. Ana"); Paciente p = new Paciente("João"); m.consultar(p); // associação via parâmetro
Agregação tem-um (fraca)

A Agregação é uma relação “tem-um” fraca. O objeto todo contém referências a objetos parte, mas esses objetos parte podem existir independentemente do todo. São criados fora do container e passados a ele.

Exemplo: uma Turma tem Alunos. Se a turma for encerrada, os alunos continuam existindo.

public class Turma { private String nome; private List<Aluno> alunos = new ArrayList<>(); // Aluno criado FORA da Turma — agregação public void addAluno(Aluno a) { alunos.add(a); } } Aluno a1 = new Aluno("Carlos"); Aluno a2 = new Aluno("Maria"); Turma t = new Turma("Java 101"); t.addAluno(a1); t.addAluno(a2); // Se Turma for destruída, Alunos continuam existindo!
Composição tem-um (forte)

A Composição é uma relação “tem-um” forte. Os objetos parte não podem existir sem o todo — são criados pelo container e compartilham o mesmo ciclo de vida. Se o todo for destruído, as partes também são.

Exemplo: um Pedido tem ItemPedido. Se o pedido for cancelado, seus itens não fazem sentido isoladamente.

public class Pedido { private List<ItemPedido> itens = new ArrayList<>(); public void addItem(String produto, double preco) { // ItemPedido criado DENTRO do Pedido — composição itens.add(new ItemPedido(produto, preco)); } } Pedido p = new Pedido(); p.addItem("Teclado", 150.0); p.addItem("Mouse", 80.0); // Se Pedido for destruído, ItemPedidos também são!

Quando Usar Cada Relação? Boas Práticas

A escolha impacta diretamente na manutenibilidade e no design do sistema:

◆ Composição — use quando a parte NÃO faz sentido fora do todo. Ex: Coração em Pessoa, ItemPedido em Pedido.
◇ Agregação — use quando a parte PODE existir sem o todo. Ex: Aluno em Turma, Professor em Escola.
→ Associação — use quando classes colaboram pontualmente, sem posse. Ex: Médico atende Paciente.
A Classe Abstrata — use quando subclasses compartilham código comum mas cada uma tem sua própria implementação de certos comportamentos.

Princípio GoF: “Prefira composição à herança” — composição promove menor acoplamento e maior flexibilidade.

🚀 Curiosidades sobre Composição e Classes Abstratas
🏗
“Prefira composição à herança” — GoF
O livro Design Patterns (Gang of Four) recomenda preferir composição sobre herança. Herança cria acoplamento forte entre classes — mudar a superclasse pode quebrar as subclasses. Composição é mais flexível: você pode trocar a implementação das partes sem afetar o todo.
📐
Notação UML para cada relação
Em diagramas UML: Associação = seta simples (→). Agregação = losango vazio (◇—) na ponta do todo. Composição = losango cheio (◆—) na ponta do todo. Saber ler UML é fundamental para comunicar design em equipe.
🗃
JPA e as relações em bancos de dados
No JPA (Java Persistence API), composição é mapeada com @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true) — os itens são deletados junto com o pai. Agregação usa @ManyToMany ou @OneToMany sem cascade, pois as partes persistem de forma independente.
⚡
Classe Abstrata vs Interface — quando escolher?
Use classe abstrata quando as subclasses compartilham código comum (atributos e métodos concretos) e representam uma hierarquia real “é-um” (Cachorro é Animal). Use interface quando quiser definir apenas um contrato de comportamento “pode-fazer” (Voavel, Serializable) para classes sem relação hierárquica.
🎨
Padrão Decorator usa composição
O padrão de projeto Decorator (GoF) é um exemplo clássico de composição. Um CafeLatte contém um Cafe e adiciona comportamento sem alterar a classe original. Streams do Java IO (BufferedReader envolve FileReader) são implementados com Decorator.
🔗
Coesão alta + acoplamento baixo
Composição favorece baixo acoplamento: classes dependem de interfaces/contratos, não de implementações concretas. Combine com o Princípio da Responsabilidade Única (SRP) — cada classe faz uma coisa bem — para código manteniível e testável.

O que é uma Exceção? Throwable

Uma exceção é um evento inesperado que ocorre durante a execução do programa e que pode comprometer seu funcionamento normal. Em vez de o programa travar abruptamente, Java permite capturar e tratar esses eventos de forma controlada.

✗ ArithmeticException — divisão por zero.
✗ ArrayIndexOutOfBoundsException — acesso a um índice inválido em um array.
✗ NullPointerException — uso de uma variável não inicializada (null).
✗ IOException — erro na leitura de arquivos.
Hierarquia de Exceções em Java Throwable

Em Java, todas as exceções são objetos da classe Throwable, que possui duas subclasses principais.

Throwable ├─ Error // grave, geralmente irrecuperável (ex: OutOfMemoryError) └─ Exception // condições que o programa PODE tratar ├─ RuntimeException // unchecked — ex: NullPointerException └─ Other Exception // checked — ex: IOException, SQLException
Error — não tratável Exception — tratável

Referência: linkedin.com/pulse/tratamento-de-exceptions-java-douglas-dias

Checked vs Unchecked Exceptions Comparação

As exceções se dividem em dois grandes grupos quanto ao momento de verificação.

CaracterísticaChecked (Verificada)Unchecked (Não Verificada)
VerificaçãoTempo de compilaçãoTempo de execução
TratamentoObrigatório (try-catch ou throws)Opcional (recomendável)
SuperclasseException (exceto RuntimeException)RuntimeException
ExemplosIOException, SQLExceptionNullPointerException, ArithmeticException, ArrayIndexOutOfBoundsException
Bloco try-catch tentar / pegar

O bloco try-catch captura a exceção e executa um código alternativo, evitando que o programa trave.

TRATANDO UMA IOException
try { FileReader arquivo = new FileReader("dados.txt"); } catch (IOException e) { System.out.println( "Erro ao ler o arquivo: " + e.getMessage()); }
TRATANDO UMA ArithmeticException
try { int resultado = 10 / 0; } catch (ArithmeticException e) { System.out.println( "Erro: divisão por zero."); }
Palavras-chave throw e throws lançar

throw lança manualmente uma exceção. throws indica, na assinatura do método, que ele pode gerar exceções — obrigando quem o chama a tratar essa possibilidade.

public static void verificarIdade(int idade) throws Exception { if (idade < 18) { throw new Exception("Idade inválida! Deve ser maior que 18."); } } // Quem chama o método precisa tratar a exceção try { verificarIdade(15); } catch (Exception e) { System.out.println("Erro: " + e.getMessage()); } // Saída: Erro: Idade inválida! Deve ser maior que 18.
throw — lança throws — declara
Exceção Personalizada extends Exception

Podemos criar nossas próprias exceções para tratar erros específicos do sistema, estendendo Exception e repassando a mensagem ao construtor da superclasse com super(mensagem).

MinhaExcecao.java
public class MinhaExcecao extends Exception { public MinhaExcecao( String mensagem) { super(mensagem); } }
USO — validarNota
static void validarNota(int n) throws MinhaExcecao { if (n < 0 || n > 10) throw new MinhaExcecao( "Nota inválida!"); } try { validarNota(15); } catch (MinhaExcecao e) { System.out.println( "Erro: " + e.getMessage()); }

🚀 Curiosidades sobre Exceções
💣
Por que não capturar Error?
Embora seja tecnicamente possível fazer catch (Error e), isso é considerado má prática. Erros como OutOfMemoryError e StackOverflowError indicam que a JVM está em um estado instável — tentar continuar a execução normalmente pode causar mais danos do que deixar o programa encerrar.
🧹
O bloco finally sempre executa
Além de try-catch, Java oferece o bloco finally, que executa sempre — com ou sem exceção, e até mesmo se houver um return dentro do try. É o lugar ideal para fechar recursos como arquivos e conexões de banco de dados.
🔌
try-with-resources
Desde o Java 7, recursos que implementam AutoCloseable (como FileReader) podem ser declarados dentro dos parênteses do try — o Java fecha o recurso automaticamente ao final do bloco, sem precisar de finally: try (FileReader f = new FileReader("a.txt")) { ... }.
⛓
Exception chaining (causa raiz)
É possível encadear exceções passando a exceção original como causa: throw new MinhaExcecao("Falha", causaOriginal). Isso preserva o stack trace completo, essencial para depurar erros em sistemas com múltiplas camadas.
💸
Exceções têm custo de performance
Criar uma exceção em Java captura o stack trace inteiro, o que tem custo computacional. Por isso, exceções não devem ser usadas para controle de fluxo normal (como substituir um if) — apenas para situações realmente excepcionais.
🏛
Checked exceptions são polêmicas
Linguagens modernas como Kotlin e C# não possuem exceções checked — todas são tratadas como unchecked. A obrigatoriedade do Java em declarar throws é vista por muitos desenvolvedores como verbosa, embora ajude a documentar contratos de métodos.
